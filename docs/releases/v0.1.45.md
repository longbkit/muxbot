# clisbot v0.1.45

Released: 2026-04-27

`v0.1.45` is the official public release after `v0.1.43`. `v0.1.44` was kept as an internal pre-release branch and was not published externally.

## In One Line

`clisbot` now has a cleaner team routing and auth model, explicit app-centered timezone control for wall-clock schedules, safer automatic config upgrade, clearer AI-discoverable CLI help, compact sender/surface prompt context, and fully compatible Gemini routing support.

## Highlights

### 1. Shared group access is easier to reason about

- CLI route ids now standardize on `group:<id>`, `group:*`, `dm:<id>`, `dm:*`, and `topic:<chatId>:<topicId>`.
- Stored bot config uses `directMessages` and `groups` with raw ids plus `*`, so config files stay provider-scoped and less noisy.
- Slack `channel:<id>`, `groups:*`, and older wildcard forms remain accepted as compatibility input from released `0.1.43` configs and older CLI habits.

### 2. Admission policy and sender policy are separated

- `groupPolicy` and Slack `channelPolicy` decide whether a multi-user surface is admitted.
- `groups["*"].policy` and exact `groups["<id>"].policy` decide who can speak inside admitted groups.
- Adding a group route under the default `allowlist` admission policy now makes that group usable immediately with sender policy `open`.
- `disabled` means fully silent on that surface, including owner/admin and pairing guidance.

### 3. Allow and block users now work for real team use cases

- `allowUsers` and `blockUsers` are enforced before the runner sees a shared group prompt.
- `group:*` can allow or block a user across every admitted group under a bot.
- Owner/admin do not bypass group admission, but they can bypass sender allowlists after a group is admitted; `blockUsers` still wins.
- Route CLI help now includes concrete allow/block examples for groups, DMs, wildcard defaults, Slack, and Telegram.

### 4. Timezone control is explicit and app-centered

- Added `clisbot timezone get|set|clear|doctor` for the app-wide default timezone.
- Added scoped `get-timezone`, `set-timezone`, and `clear-timezone` commands for agents, routes, and concrete bots.
- `routes get-timezone` prints the route override, effective resolved timezone, and current local time for that surface so bootstrapped agents can ask users to confirm without understanding the fallback chain.
- New default workspace bootstrap guidance asks agents to confirm timezone and local time before writing timezone notes.

### 5. Wall-clock loops are safer

- CLI wall-clock loop creation now confirms the first schedule before persisting it, showing the resolved timezone, next run, and exact retry command with `--confirm`.
- Calendar loops freeze the resolved effective timezone on the loop record so later timezone config changes do not silently move existing schedules.
- Chat `/loop` creation stays low-friction, but created-loop replies include timezone, local next run, UTC next run, cancel command, and correction guidance.
- Agent prompt guidance now tells AI agents to inspect `clisbot loops --help` for schedule, loop, and reminder requests instead of guessing loop CLI behavior.
- Codex-backed loop runs now wait for the interactive prompt marker before submitting the first scheduled prompt, avoiding "loop is now running" notifications followed by an unsubmitted prompt in a fresh tmux pane.
- Existing live tmux sessions without a stored CLI `sessionId` are reused directly instead of injecting a surprise `/status` command before the next user prompt.
- CLI loop creation now requires `--sender <principal>` and stores the creator metadata on persisted loops, preventing agent-created CLI loops from later rendering scheduled prompts with sender unavailable.

### 6. Config upgrade is automatic and visible

- On first read of a released `0.1.43` config, clisbot backs up the original file under `backups/`.
- It then migrates directly to schema `0.1.45`, validates the new shape, writes the upgraded config, and logs each stage.
- Internal pre-release `0.1.44` configs also upgrade to `0.1.45`.
- If the config is already on the current schema, the upgrade path is skipped without backup or upgrade logs.
- Legacy timezone defaults are migrated into `app.timezone`, then removed from default-level legacy fields.
- Runner startup heuristics that should move with clisbot releases are no longer pinned into generated config by default.
- During upgrade, `0.1.45` clears the persisted Codex `startupReadyPattern`, Gemini startup handshake fields, and default runtime monitor backoff ladder so those values can inherit the release-owned defaults again.

### 7. Gemini compatibility status is corrected

- README now marks Gemini as fully compatible for routed Slack and Telegram workflows.
- The previous "usable with caveats" label was too conservative for the current runner support.

### 8. Base workspace templates are decoupled from OpenClaw references

- Runtime bootstrap now reads base markdown templates from `templates/default`.
- `templates/openclaw` is no longer used by start/init/agent bootstrap logic.
- This makes it safe to update or retain `templates/openclaw` as reference material without changing generated clisbot workspaces.

### 9. Telegram voice and audio messages reach agents

- Telegram `voice` and `audio` payloads are downloaded into the agent workspace attachment directory.
- Audio paths are passed as normal `@file` mentions, matching image and document attachments so multimodal CLIs can resolve them consistently.
- Attachment storage now maps common audio content types to stable extensions and uses Telegram metadata when file downloads report a generic binary content type.

### 10. Agent prompts now use sender and surface context

- Channel prompts now render compact `Message context` with `time`, `sender`, and `surface` instead of older mixed identity prose.
- Slack prompts opportunistically enrich that context with `users.info` and `conversations.info`, falling back to stable ids if lookup is unavailable.
- The Slack app manifest now includes `users:read`, `channels:read`, `groups:read`, `im:read`, and `mpim:read` so new apps can show readable sender and surface names.
- Telegram topic prompts use available topic-creation metadata and the surface directory to show topic display names when the current message payload does not carry them.
- Surface directory updates preserve known display names and handles when a later provider event only carries ids.
- Steering, queue, and loop delivery rebuild prompt context through the same contract so long-running and delayed work does not use stale prompt envelopes.
- Prompt guidance includes the read-only `clisbot auth get-permissions --sender <principal> --agent <id> --json` check before sensitive actions or clisbot configuration changes.
- Permission JSON returns effective `allowed` plus `explanation` by default, including `runShellSlashCommand` for clisbot slash-command shell access.
- Surface directory records store `senderId`, `surfaceId`, provider ids, display names, and handles without storing prompt-rendered text or CLI target syntax.

## Upgrade Notes

- Config schema is now `0.1.45`.
- Public upgrade path is `0.1.43 -> 0.1.45`; `0.1.44` was not publicly released.
- Existing `0.1.43` configs are upgraded automatically on first read.
- A backup is written before the upgraded config is applied, for example:

```text
~/.clisbot/backups/clisbot.json.0.1.43.<timestamp>
```

- After the config is rewritten to `0.1.45`, downgrading to `0.1.43` should use that backup.
- Dedicated downgrade and restore CLI UX is tracked separately in the backlog.
