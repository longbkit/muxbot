# clisbot v0.1.43

Released: 2026-04-21

`v0.1.43`은 runtime recovery를 더 durable하게 만들고, routed follow-up control을 더 분명히 하며, tmux prompt submission check를 더 truthful하게 만들고, non-streaming message-tool 흐름에서 queued-start notification을 되살리고, Slack thread reply가 root message의 오래된 attachment를 상속하는 문제를 막습니다.

## 한 줄 요약

`clisbot`은 이제 runtime / tmux startup trouble에서 더 정직하게 복구하고, routed conversation의 follow-up control을 더 분명히 보여 주며, prompt / attachment 전달 경계를 더 엄격히 해 실제 부하에서도 Slack과 Telegram session이 더 올바르게 동작합니다.

## 하이라이트

### 1. Runtime recovery가 더 부드럽고 덜 영구적으로 멈춤

- runtime monitor 기본 ladder가 1분, 3분, 5분, 10분, 15분, 30분으로 완만하게 증가
- 최종 단계에 도달해도 완전히 포기하지 않고 계속 retry
- Telegram polling conflict도 runtime 안에서 복구

### 2. Routed follow-up control이 더 명확

- `/mention`은 `/followup mention-only`의 빠른 shorthand
- `/mention channel`, `/mention all`은 적절한 scope에 mention-only 기본값을 저장하고 현재 대화에도 바로 적용
- `/transcript`는 compact recent session snapshot을 기본으로 반환

### 3. tmux prompt delivery truthfulness 강화

- submit confirmation이 paste가 보인 뒤의 pane snapshot을 기준으로 동작
- 보이는 paste를 성공적인 Enter submit으로 오인하는 false positive 경로를 막음

### 4. queued message가 실제 시작 시점을 더 정확히 알림

- `message-tool` + `streaming: off`에서도 첫 `running` update에서 queued-start notification을 보낼 수 있게 됨

### 5. Slack thread reply가 root thread attachment를 상속하지 않음

- 현재 inbound message에 `files`가 없을 때도 root thread의 오래된 파일을 끌고 오지 않음
